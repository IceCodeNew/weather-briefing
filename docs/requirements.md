# 产品需求

本文只说明产品要解决的场景和应有行为。部署方法见 [README](../README.md)，实现方式见[系统设计](design.md)。

## 目标

用户希望在固定时间收到当天预报，并在天气发生值得关注的变化时及时收到提醒。简报需要整合多个来源、保留前后文，并避免重复发送没有行动价值的信息。

## 每日预报

- 默认每天 08:00 生成当天预报，时间可以调整。
- 预报包含天气、气象预警和必要的自然灾害信息。
- 预报给出穿衣、除湿、运动和口罩建议。
- 有过敏指数或花粉信息时，还要给出过敏建议。
- 空气质量必须注明指数标准。
- PM2.5 只有在来源直接给出浓度和单位时才展示。
- 用户可以查看当地今天或未来某一天的预报。
- 查询未来日期时，不得用当前观测冒充未来预报。

## 变化提醒

- 默认在 09:00–23:00 每小时检查一次，时间范围可以调整。
- 只有用户可能需要采取行动时才发送消息。
- 值得提醒的变化包括临近降雨、明显温度或风力变化，以及预警变化。
- 确实影响关注地点的灾害动态也应提醒。
- 普通天气复述、轻微波动和没有变化的持续预警不应打扰用户。
- 未发送的信息要保留，并在后续判断中与新信息一起考虑。
- 每天最后一次检查时，如果当天还没有发送过变化提醒，则发送一条无声消息。
- 用户手动运行时，应立即发送积压信息，不受调度窗口限制。

## 内容要求

- 标题直接概括最重要的天气情况，不再重复设置摘要段落。
- 正文按天气、预警、灾害动态和生活建议组织。
- 变化提醒不重复每日生活建议。
- 同一事实不得在标题和各章节之间反复出现。
- 标题和每条结论都要附上本轮输入中的来源链接。
- 来源名称要让用户能识别，链接要能核验对应内容。
- 同名同链接的来源只显示一次，不同链接不得误合并。
- 简报语言由地点配置决定，默认是英文。
- 来源本身已经使用目标语言时，不应做无意义的往返翻译。

## 地点与地区

- 用户可以配置一个或多个地点，各地点的历史和状态必须隔离。
- 每个地点都必须有唯一 ID，并至少提供以下两项之一：地点名称；完整的经纬度坐标。
- 只有地点名称时，程序应找到坐标，并把精确匹配的坐标补充到地点配置中供以后运行复用。
- 只有经纬度坐标时，程序应找到便于阅读的地点名称和行政区信息，并把缺失的地点名称补充到地点配置中。
- 地点名称和经纬度坐标都有时，不应再请求定位服务。
- 名称过于具体而无法识别时，可以逐步降低精度。
- 首次使用较低精度匹配时，要把匹配结果发给用户确认。
- 较低精度匹配在用户确认前不能自动写入地点配置。
- 中国大陆、新加坡和日本应使用适合当地的天气信息。
- 同一时间和地区的信息冲突时，应优先采用当地权威气象机构的最新资料，并保留冲突来源供用户核验。
- 用户也可以明确指定天气来源及备用顺序。
- 本地天气来源缺少完整数据时，可以与全球天气来源组合使用。

## 信息来源

- 天气服务是主要来源，订阅源是可选补充。
- 没有配置可选来源，或可选来源暂时失败时，天气简报仍应运行。
- 每轮只读取按地点时区判断为当天发布的文章。
- 已处理文章不能重复总结。
- 用户指定的权威预报文章要清理页面噪声后完整转发，不做摘要或截断。
- 清理后没有正文的文章不能发送，也不能标记为已处理。
- 新简报要参考历史简报、未发送信息、权威预报和当前有效预警。
- 较新的资料取代旧资料后，旧结论不能继续出现在当前简报中。

## 预警与灾害

- 预警不能因为一段时间没有再次出现就自动消失。
- 只有明确解除、降级或超过保留时间时，才更新预警状态。
- 无效的预警解除信息不能删除现有状态，也不能阻断整份简报。
- 只跟踪明确影响关注地点或其上级行政区的灾害。
- 位于远处、已经移出或明确无影响的灾害不得通知。

## 可靠性

- 可选来源失败不能阻断主要天气流程。
- 主要任务失败、来源连续失败或长期没有更新时，要发送运维告警。
- 告警投递失败后应继续重试，但不能覆盖原始业务错误。
- 历史输入受限时，至少保留最新值和比较基线。
- 关键状态必须一起保存，不能留下只更新一半的结果。
- 独立转发的正文失败后要继续投递，已经确认成功的正文不能重发。
- 外部平台已经接收但本地尚未确认时，允许出现少量重复，不能静默丢失。

## 隐私与运维

- 公开仓库不能包含凭据、真实地点、私密 URL、历史简报或运行状态。
- 常规日志不能记录凭据、坐标、正文、私密 URL 或接收方标识。
- 调试日志可以记录非敏感的请求结果、耗时和处理阶段。
- 完整正文诊断必须默认关闭、限时启用，并明确提示隐私风险。
- 所有时间都必须带明确时区。
- 运行状态必须持久保存，服务重启后不能丢失记忆。

## 运行与投递

- 用户可以让服务长期运行并按计划发送简报，也可以手动生成一次预报或变化提醒。
- 首个投递平台是 Telegram，后续可以增加其他投递平台。
