# 技术取舍

本文只解释乍看不够理想、但目前有意保留的选择。正常的模块拆分和通用实现不在这里重复。当前实现见[系统设计](design.md)。

每项都说明选择、成立条件和重新评估的时机。

## 完整天气服务仍返回多种能力

天气能力已经分开建模，但和风天气和 Open-Meteo 仍通过一次完整流程返回天气、空气质量、生活指数或过敏原。

为了类型上的纯粹而拆成多次请求，会增加延迟、配额消耗和部分失败状态。因此完整服务目前放在天气槽位，只有真正独立的服务才进入其他能力槽位。

这项选择成立的条件是厂商接口仍适合一次组合调用。如果某项能力可以独立调用，并且确实需要单独替换或路由，就增加对应的薄适配器。

`ALLERGEN` 只表示结构化过敏原数据，不表示任何过敏建议。和风天气的综合过敏指数属于生活指数，即使它会触发过敏建议。

如果厂商以后提供具体花粉种类和浓度，再把它声明为结构化过敏原能力。

## 默认简报语言是英文

简报语言以前是隐含行为。变成配置项后，产品级默认值选择英文，现有中国大陆示例则显式填写 `zh-CN`。

这样不会把一个地区的旧行为误设为全世界的默认值，也能让旧部署通过明确配置保留原有输出。

如果产品以后有可靠的用户语言来源，可以重新评估是否在首次配置时提供建议值。程序不能只根据坐标猜测用户想读的语言。

## 简报语言不写入定位缓存

地点语言是用户当前选择，不是定位服务返回的事实。因此缓存只保存坐标、行政区和时区；每次运行都从地点配置覆盖简报语言。

代价是所有缓存命中路径都必须执行覆盖。如果语言开始影响定位服务返回的名称或行政区，就把语言加入缓存键，而不是继续复用当前缓存。

## 只支持基础 BCP 47 标签

当前内置服务和地点配置只需要语言、脚本和地区部分。完整支持 Unicode 扩展和私有子标签需要引入成熟解析库，当前没有实际收益。

因此实现只接受基础标签，并按主语言逐步匹配服务支持的输出语言。

如果外部服务、持久化格式或用户配置开始需要扩展子标签，就改用成熟的 BCP 47 解析器，并重新检查规范化和匹配规则。

## 保留旧 DeepSeek 环境变量

项目在引入 any-llm 之前直接读取 `DEEPSEEK_MODEL` 和 `DEEPSEEK_BASE_URL`。改用 any-llm 后，通用模型名称改由 `LLM_MODEL` 提供，DeepSeek 的 API Base 则使用 SDK 约定的 `DEEPSEEK_API_BASE`。

为了不让已有部署在升级时立即失效，配置入口仍接受旧变量。只有对应的新变量没有填写时，旧变量才作为后备；新旧变量同时存在时始终采用新变量。

这是针对历史 DeepSeek 部署的定向兼容，不是为所有模型厂商维护第二套配置名称，也不能扩展成通用的 provider 配置注册表。

如果以后发布明确允许删除旧配置的重大版本，并且迁移说明已经给已有部署留出足够时间，就可以移除这两个后备变量。在此之前，修改模型配置边界时必须保留并测试这一优先级。

## 模型对象的关闭使用兼容探测

any-llm 当前没有统一的关闭接口。应用先尝试 provider 的 `aclose()` 或 `close()`，再清理它直接持有的已知 SDK 资源。

这个做法对对象布局有有限依赖，但能避免应用创建的网络资源泄漏。关闭失败只记录安全的类型信息，不能覆盖已经发生的业务异常。

any-llm 提供稳定的统一生命周期接口后，应删除属性探测并直接使用该接口。

## 未知预警解除 ID 不阻断简报

模型可能根据历史内容返回已经过期或不存在的预警 ID。这些 ID 不会删除任何状态，也不会出现在用户正文中。

应用在保存前过滤未知 ID，只记录不同 ID 的数量，避免为没有副作用的记账错误消耗一次模型修复并丢弃整份简报。

如果解除字段开始影响正文或触发外部操作，就恢复严格失败，并先把预警身份交给确定性的服务或应用逻辑管理。

## 历史预算目前只限制天气快照

天气 API 快照每次调度都会产生，是历史输入中稳定增长的部分。RSS 文章和已发布简报目前频率较低，仍只受时间窗口限制。

因此文档数和字符数预算目前只应用于历史天气快照。应用记录完整模型输入的字符数，但不记录正文。

如果 RSS 数量、地点数或简报频率明显增加，或者持续出现输入过长和模型超时，就改成覆盖所有历史类别的统一预算。

## 快照额外保存可比较值

面向用户的天气文本包含更新时间等易变内容，不能直接判断天气是否真的变化。各适配器因此提供 `history_value`，只排除不会改变结论的元数据。

这会在 SQLite 中多保存一份确定性表示，但避免通过解析渲染文本来猜测语义，也让预算告警可以稳定去重。

如果适配器难以维护稳定值、经常漏掉重要字段，或重复存储成为容量问题，就改用带版本的结构化值或摘要指纹，并设计迁移规则。

## 不用模型压缩单份超长历史快照

在主调用前增加一次模型摘要会增加费用、超时和故障面，也无法保证预警、空气质量和过敏原信息不丢失。

内置适配器改为从已校验的领域对象生成较短的确定性摘要。当前轮仍使用完整上下文，摘要只用于历史比较。

最新值和窗口基线即使摘要后仍无法容纳时，应用继续生成简报并发送运维告警。如果确定性摘要长期无法满足实际质量，再评估可缓存、可观察且与主调用隔离的压缩方案。

## 主简报确认后再保存结果

主简报和完整转发正文不是同一个平台操作。主简报被平台接受后，应用立即保存本轮结果，并把待发正文登记到同一事务中。正文随后逐篇投递和确认。

这样正文中途失败不会让下一轮重新生成主简报，也不会重发已经确认的前项。但平台接受消息后、本地确认前崩溃，仍可能造成重复。

当前接受少量重复，因为 Telegram 没有可复用的幂等键，静默丢失比重复更糟。

如果平台提供稳定消息身份、重复开始造成实际问题，或需要多个 worker 并发投递，就把幂等键、租约和确认协议放到投递适配器中。

## 异步流程仍使用同步 SQLite 和文件操作

状态库和定位缓存直接在事件循环线程中执行短小的本地操作，没有引入异步 SQLite 依赖或 `asyncio.to_thread`。

当前任务按小时运行、地点按顺序处理，状态量受保留窗口限制。正常的本地磁盘操作远短于天气、RSS、定位和模型网络请求。

这项选择要求状态文件位于低延迟本地磁盘或同等持久卷，并且没有持续的事件循环停顿、锁等待或调度漂移。

如果地点开始并行处理、状态迁移到网络存储、写入量明显增加，或监控发现阻塞，就测量延迟并重新设计持久化边界。

不能简单把现有连接包进 `asyncio.to_thread`，因为 SQLite 连接具有线程亲和性。

## 第三方 DEBUG 日志保持关闭

模型和 HTTP SDK 的 DEBUG 内容不受应用控制，可能包含凭据、请求地址、正文或位置上下文。因此 `DEBUG=true` 只提高应用 logger 的级别，第三方 logger 仍保持 WARNING。

完整正文诊断只记录应用自己控制的字段，并要求额外的限时开关。

如果第三方 SDK 以后提供稳定的结构化安全日志接口，可以评估接入；不能依赖异常消息或事后正则清洗来保护隐私。

## Telegram 投递继续直接调用 Bot API

Telegram 投递目前只调用 Bot API 的 `sendMessage`。项目继续复用已有的 HTTPX 客户端、统一网络生命周期和隐私安全日志，不为这一个端点引入 `python-telegram-bot` 等完整框架。

Telegram publisher 只负责构造请求、平台长度限制、HTML 分块和安全错误分类。它不实现更新轮询、Webhook、会话状态或 Bot 命令路由。

如果投递开始使用多个 Telegram 端点、需要接收更新，或者 Bot API 兼容工作持续增加，就重新评估维护活跃的 SDK，并删除能由 SDK 可靠承担的自定义协议代码。
